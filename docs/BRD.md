# SmartCart PVM — Business Requirements Document

**Версия:** 1.0 | **Дата:** Март 2026 | **Статус:** Draft

---

## 1. Контекст и проблема

### 1.1 Целевой пользователь

Фитнес-тренер или спортсмен, который:

- Знает свои макро/микро-нормы, но не успевает переносить их в конкретные товары
- Не хочет ходить по магазинам ради сравнения цен
- Не хочет фотографировать штрихкоды стоя у полки
- Не может быстро понять: «этот творог за 189₽ — нормально или развод?»

**Ключевой инсайт:** проблема не в сканировании — проблема в том, что пользователь каждый раз делает одну и ту же аналитическую работу заново, вручную, в голове.

### 1.2 Персонажи

**Персонаж A: Андрей**, 29 лет, менеджер (клиент тренера)

| Параметр | Значение |
|----------|----------|
| Цель | Набрать 5 кг мышц |
| Бюджет | 4500₽/нед на еду |
| Магазины | Пятёрочка (рядом с домом), Перекрёсток (по пути) |
| Готовит | 20–30 мин вечером, по выходным нормально |
| Завтрак | Бутерброд с колбасой + кофе |
| Болит | Не добирает белок, колбасу любит и не готов отказываться |
| Не болит | Цена (средний бюджет), аллергии нет |

**Персонаж B: Игорь**, 29 лет, на сушке

| Параметр | Значение |
|----------|----------|
| Цель | Сбросить 8 кг жира, сохранить мышцы |
| Бюджет | 3000₽/нед (жёсткий) |
| Магазины | Пятёрочка, Магнит |
| Готовит | Минимально, некогда |
| Завтрак | Кофе + что попадётся |
| Болит | Дорого, лень, вечером ест пахлаву горстями |
| Ограничения | Непереносимость лактозы |

**Персонаж C: Дарья**, 34 года, фитнес-тренер

| Параметр | Значение |
|----------|----------|
| Роль | B2B-пользователь (в v2) |
| Задача | Контролировать продуктовую корзину клиентов |
| Болит | Клиенты не выполняют план питания → нет прогресса → уходят |
| Хочет | Отправить клиенту готовый список под его план питания |

### 1.3 Карта реальных болей

| # | Боль | Корневая причина | Как сейчас решают |
|---|------|-----------------|-------------------|
| P1 | Физически тяжело добирать белок | Однообразие, нет замен в привычном контексте | Протеиновые порошки — дорого |
| P2 | Дорого питаться правильно | Нет инструмента сравнения ₽/нутриент | Покупают «дешевле» = хуже состав |
| P3 | Вечером лень, думать → срыв | Decision fatigue → к концу дня нет когнитивного ресурса | Волевое усилие — не работает долго |
| P4 | Непонятно что купить вместо привычного | Замены предлагают без учёта юзкейса | Гуглят, находят несовместимые советы |
| P5 | Не могу отказаться от круассана/пахлавы | Cravings ≠ слабоволие, это нейробиология привычки | Диета → срыв → отказ от изменений |
| P6 | Не понимаю, стоит ли товар своих денег | Нет инструмента оценки value | Сравнивают цену за 100г — неверная метрика |

### 1.4 Чего продукт НЕ пытается решить

- Заменить нутрициолога или врача
- Лечить расстройства пищевого поведения
- Строить точные медицинские рекомендации без анализов
- Полностью менять пищевые привычки за неделю

## 2. Продуктовое видение

### 2.1 Формулировка

**SmartCart** убирает необходимость думать о еде каждый день — за счёт готового списка покупок, который подстраивается под привычки пользователя и тихо улучшает их.

### 2.2 Ценностное предложение

- **Для клиента тренера:** «Ты продолжаешь есть привычное — только лучшие версии того же самого»
- **Для тренера:** «Твои клиенты приходят с реальной корзиной, которая соответствует плану питания»

### 2.3 Принципы продукта

1. **Постепенность** — один своп в неделю, не ребилд корзины
2. **Юзкейс первичен** — замена только внутри одного функционального слота
3. **Честность** — показываем уровень уверенности у каждого утверждения
4. **Автопилот** — система работает без участия пользователя, он только корректирует
5. **Без запретов** — любимые продукты неприкосновенны
6. **Гибкость** — плавное погружение, пользователь контролирует темп

## 3. Функциональные требования

### RF-01: Онбординг

**Цель:** за ≤5 минут получить достаточно данных для первого свопа. Не перегружать.

**Шаги:**

**Шаг 1 — «Что тебя достало?»** (1 экран, чекбоксы)
- Варианты: не добираю белок / дорого / лень / срываюсь / не знаю что купить

**Шаг 2 — Текущие привычки по слотам** (2–3 экрана)
- Завтрак: что обычно? (бутерброд / яичница / каша / пропускаю)
- На бутерброд: колбаса / сыр / паштет / рыба
- Ужин: готовишь сам / полуфабрикаты / заказываешь

**Шаг 3 — Контекст** (1 экран)
- Цель: набор / сушка / поддержание
- Бюджет: слайдер
- Магазины: мультиселект

**Шаг 4 — Ограничения** (1 экран, опционально)
- Аллергены, нелюбимые продукты: «это никогда не трогай»

**Бизнес-правило:** система НЕ задаёт вопрос о нутритивных дефицитах без анализов. Использует популяционные данные по дефицитам для РФ (D, Mg, йод) с явным дисклеймером.

**Шаг 5 (опционально) — Фото холодильника** (см. RF-10)

### RF-02: Quality Gate

**Цель:** отфильтровать товары до того, как они попадут к пользователю.

**Алгоритм** (только доказательно обоснованные критерии):

**TIER-1 — Блокировка** (`gate_passed = false`):
- NutriScore D или E (алгоритм NutriScore-2023) [RCT]
- NOVA класс 4 И NutriScore C или ниже — одновременно [SR/MA]
- Промышленные трансжиры > порога EFSA [RCT + механистические данные]

**TIER-2 — Предупреждение** (`gate_passed = true, warning = true`):
- NOVA класс 4 при NutriScore B/C
- Эмульгаторы E466, E471–E475 в составе [RCT, короткие]
- Натрий > 1.5 г/100г

**TIER-3 — Информация** (без штрафа):
- Нитрит натрия (E250) — отображается как факт с контекстом [когорты, нет RCT]
- Прочие аддитивы — показываем без оценки

**Что система НЕ делает:**
- Не штрафует E102/E110/E124 у взрослых (данные только на детях с СДВГ)
- Не использует позицию ингредиента в составе как критерий
- Не оценивает % мяса как напрямую нутритивный показатель

**Бизнес-правило:** каждый критерий отображается с уровнем уверенности:
- ✅ `[RCT]` — подтверждено рандомизированными исследованиями
- ⚠ `[SR/MA]` — подтверждено систематическими обзорами / мета-анализами
- ℹ `[i]` — обсервационные данные, не RCT

---

### RF-03: Value Score

**Цель:** показать экономическую эффективность товара под конкретный профиль.

**Два явно разделённых блока:**

**Блок A — Нутри-оценка** `[RCT]`:
- NutriScore-2023 (алгоритм A–E) — валидирован в RCT

**Блок B — Калькулятор** (арифметика, не доказательный инструмент):
- ₽ за 1г белка
- ₽ за 100 ккал
- ₽ за единицу приоритетного нутриента из профиля
- Сравнение с медианой по слоту (cheaper / average / expensive)

**Бизнес-правило:** Блок B отображается с плашкой: «Калькулятор — полезный ориентир, не медицинская рекомендация.»

**Персонализация приоритетного нутриента в Блоке B по цели:**

| Цель | Приоритетный нутриент |
|------|----------------------|
| Набор | Белок, калории |
| Сушка | Белок, клетчатка |
| Поддержание | Белок, NutriScore |

### RF-04: Своп-система (ядро продукта)

**Цель:** предлагать замены строго внутри функционального слота.

**Таксономия юзкейсов:**

Каждый товар в базе получает два атрибута:

**convenience_tier** (INTEGER):
- 1 = ready-to-eat (открыл — ешь, 0 мин)
- 2 = quick-cook (5–10 мин, минимум действий)
- 3 = meal-base (20+ мин)

**use_context** (TEXT[]):
- `sandwich` — на хлеб, без нагрева
- `fry_with_egg` — жарить с яйцом
- `hot_meal` — горячее блюдо
- `cold_snack` — холодный перекус
- `snack` — перекус, носимый
- `breakfast_ready` — утро, быстро
- `pre_workout` — до тренировки
- `post_workout` — после тренировки
- `topping` — добавка к чему-то

**Алгоритм подбора замены:**

1. Определить `convenience_tier` текущего товара
2. Определить `use_context` текущего товара
3. Найти товары с совпадающим tier И хотя бы одним совпадающим context
4. Из них: прошедшие Quality Gate
5. Из них: отсортировать по Value Score под профиль
6. Вернуть топ-3:
   - #1 → лучший по NutriScore в слоте
   - #2 → лучший ₽/целевой нутриент в слоте
   - #3 → лучший Taste Score в слоте (если есть данные)

**Бизнес-правило:** система НИКОГДА не предлагает своп между разными `convenience_tier` без явного запроса пользователя. Замена sandwich-товара — только на другой sandwich-товар.

**Пример:**

Пользователь берёт: Колбаса Докторская Черкизово
- Слот: ready_to_eat + [sandwich, fry_with_egg]

❌ Система НЕ предлагает:
- Куриную грудку 1 кг (сырую) — другой слот (meal_base)
- Тунец в банке — другой контекст, другой вкус/запах

✅ Система предлагает:
- Колбаса куриная «Петруха» — тот же слот, лучше состав
- Нарезка индейки ВкусВилл — тот же слот, чище
- Куриное филе варёное в нарезке (вакуум) — тот же слот, +8 г белка/100г

### RF-05: Прогрессивная модель погружения

**4 этапа с триггерами перехода:**

| Этап | Название | Что предлагается | Триггер перехода |
|------|----------|------------------|------------------|
| 1 | Один своп | 1 замена в корзине | Своп куплен 2+ раза |
| 2 | Три свопа | До 3 замен параллельно | 2 из 3 свопов прижились |
| 3 | Полкорзины | Автогенерация ~50% списка | Пользователь нажал «хочу больше» |
| 4 | Автопилот | Полный список + 20–30% неприкосновенных | Пользователь включил сам |

**Бизнес-правила:**
- Откат на предыдущий этап — в любой момент, без объяснений
- Темп (медленно / комфортно / быстро) настраивается в любой момент
- «Неприкосновенные» товары — система никогда не свапает и не убирает из списка
- Нет уведомлений с давлением («ты не выполнил», «осталось 3 дня»)

**Критерии «зацепленности» для перехода:**

| Переход | Триггер |
|---------|---------|
| 1 → 2 | Первый своп куплен 2+ раза |
| 2 → 3 | Хотя бы 2 из 3 свопов прижились (покупка 2+ раз) |
| 3 → 4 | Пользователь сам нажал «собрать автоматически» |
| Откат назад | В любой момент, без объяснений |

**Тон сообщений по этапам:**

| Этап | Тон | Пример |
|------|-----|--------|
| 1 | Нейтральный, без обязательств | «Вот вариант — попробуешь?» |
| 2 | Лёгкое поощрение факта | «Куриная грудка уже третью неделю в списке» |
| 3 | Предложение расширить | «Хочешь, возьмём больше на себя?» |
| 4 | Партнёрский | «Список готов. Твои любимые продукты на месте.» |

Никаких стриков, бейджей, «ты молодец».

### RF-06: Taste Engine (коллаборативная фильтрация)

**Цель:** улучшать релевантность рекомендаций через поведенческие сигналы.

**Заявленная роль:** инструмент retention и удобства, НЕ медицинской персонализации.

**Сигналы и веса:**

| Событие | Вес |
|---------|-----|
| Добавил в список | +0.3 |
| Купил (подтверждён чек/повторение) | +0.7 |
| Купил повторно 2+ раз | +1.5 |
| Явный лайк | +1.0 |
| Удалил из списка | −0.5 |
| Явный дизлайк / «не моё» | −1.0 |
| Проигнорировал рекомендацию 3 раза | −0.3 |

**Алгоритм:** гибридный SVD + content-based + NutriScore re-ranking.

**Холодный старт:**
- 0–10 взаимодействий → только Quality Gate + Value Score
- 10–50 → подключается item-based CF
- 50+ → полный гибридный режим (user-based с косинусным сходством)

**Контекстные кластеры** (матрица строится внутри кластеров):
- По цели (набор / сушка / поддержание)
- По бюджетному диапазону (±30%)
- По ограничениям (без лактозы / без глютена)
- По региону

**Холодный старт нового товара:**
- Первые 20 дней — только Quality Gate + Value Score
- После 30 покупок разными пользователями — входит в матрицу CF

### RF-07: Список покупок

**Генерация:** каждое воскресенье автоматически + по запросу.

**Структура списка:**
- **Шапка:** Период / Итого ₽ / Прогресс по белку и калориям
- **Секции** по категориям товаров
- **Карточка товара:** Название · Магазин · Цена / NutriScore · ₽/г белка (если релевантно) / Тег: «Автоподобрано» / «Моё постоянное»
- **Своп недели** — выделенный блок
- **Футер:** Суммарно Белок Хг / Жиры Хг / Углеводы Хг / ⚠ Дефицит (если есть)

**Коррекция:**
- Свайп влево → замены в том же слоте (топ-3)
- Тап → детальная карточка товара
- «Другой магазин» → аналог в другом магазине из профиля

---

### RF-08: Planned Indulgence

**Цель:** не запрещать любимые продукты, а встраивать их в план.

**Механика:**
- Пользователь добавляет «неприкосновенные» продукты в профиле: кофе, круассан, пахлава
- Система учитывает их калорийность и БЖУ при генерации списка
- Остаток компенсируется в других приёмах того же дня
- Никаких «ты съел лишнее» — только тихая компенсация

**Бизнес-правило:** система НИКОГДА не показывает негативную оценку «запрещённых» продуктов. Только нейтральный факт, если пользователь сам спросит.

---

### RF-09: Дисклеймеры и уровни уверенности

Обязательно присутствуют на каждом экране с нутритивными утверждениями.

**В карточке товара:**
- ℹ NutriScore основан на алгоритме NutriScore-2023, валидирован [RCT]
- ℹ Оценка состава по NOVA и данным EFSA [SR/MA]
- ₽/г белка — калькулятор, не медицинская рекомендация

**В онбординге:**
- «Рекомендации по нутриентам основаны на популяционных данных ВОЗ/Роспотребнадзора. Для персональных рекомендаций — сдайте анализы.»

---

### RF-10: Фото холодильника (стартовый инвентарь)

**Цель:** определить что уже есть дома, не добавлять в список.

**Поток:**
1. Пользователь делает 2–4 фото: дверца холодильника, полки, шкаф с крупами
2. Распознавание через Vision AI (fine-tuned YOLOv8 / LLM Vision на российских продуктах)
3. Пользователь видит список распознанного, подтверждает/правит/убирает
4. Подтверждённые товары НЕ попадают в список на эту неделю (или с пометкой «уже есть»)

**Технические требования:**
- Обработка ≤10 секунд
- Фото удаляется с сервера сразу после распознавания (приватность)
- Confidence threshold <70% → товар в раздел «для подтверждения»
- Шаг полностью опциональный — не блокирует онбординг
- Офлайн вариант: ручной ввод через поиск

## 4. Бюджетные сегменты

Разные бюджеты = разная логика, не один сценарий.

| Сегмент | Бюджет/нед | Реальная проблема | Приоритет в Value Score |
|---------|-----------|-------------------|------------------------|
| Жёсткий | <2500₽ | Купить достаточно белка вообще | Максимум нутриента за рубль |
| Средний | 2500–5000₽ | Не переплачивать, баланс цена/качество | Value Score: сравнение в слоте |
| Комфортный | 5000–9000₽ | Вкусное и качественное, не тратить время | Taste Engine + NutriScore A/B |
| Премиум | >9000₽ | Оптимальный состав, минимум компромиссов | NutriScore A/B, органика, ВкусВилл |

**Калибровка весов Value Score по бюджету:**

| Бюджет | w_nutriscore | w_price | w_deficit |
|--------|-------------|---------|-----------|
| <3000₽ | 0.25 | 0.55 | 0.20 |
| 3000–6000₽ | 0.35 | 0.35 | 0.30 |
| >6000₽ | 0.45 | 0.15 | 0.40 |

## 5. Use Cases

### UC-01: Первый визит (нет аккаунта)

**Актор:** Андрей (Персонаж A)
**Предусловие:** приложение установлено, аккаунта нет

**Основной поток:**
1. Видит экран «Что тебя достало в питании?» — выбирает «не добираю белок» и «не знаю что покупать»
2. Отвечает на вопросы о завтраке: бутерброд с колбасой
3. Указывает цель (набор), бюджет (4500₽), магазины (Пятёрочка, Перекрёсток)
4. Видит прогресс-бар «Анализирую 2800+ товаров...»
5. Получает экран с одним свопом: Докторская → Куриное варёное в нарезке
6. Видит сравнение: то же использование (бутерброд, 0 мин), +30г белка, +10₽
7. Нажимает «Добавить в список»
8. Видит список из привычных товаров + один новый

**Альтернативный поток (7a):** нажимает «Не сейчас» → список генерируется без свопа, предложение повторится через неделю

**Постусловие:** профиль создан, первый список сгенерирован, своп предложен

### UC-02: Вечерний поход в Пятёрочку

**Актор:** Игорь (Персонаж B), после работы, усталый
**Предусловие:** список на неделю уже сгенерирован в воскресенье

**Основной поток:**
1. Открывает приложение у входа в магазин
2. Видит список — 8 позиций, отфильтрованы под Пятёрочку
3. Идёт по списку, ставит галочки
4. На позиции «Творог» видит: ⚠ «Закончился — вот замена в том же слоте»
5. Система предлагает 2 аналога: тот же cold_snack, тот же convenience_tier, тот же контекст
6. Выбирает первый, ставит галочку

**Альтернативный поток (4a):** товар есть, но цена выросла на 25%+ → система показывает «Подорожал. Вот аналог дешевле.»

**Постусловие:** покупки сделаны, взаимодействия записаны в user_product_interactions

### UC-03: Своп не подошёл

**Актор:** Андрей
**Предусловие:** система предложила своп недели (куриное варёное в нарезке)

**Основной поток:**
1. Андрей купил товар на первой неделе
2. На второй неделе свайпает его влево → «Не моё»
3. Видит вопрос «Что не так?» — варианты: дорого / вкус не понравился / неудобная упаковка / просто не хочу
4. Выбирает «Вкус не понравился»
5. Система возвращает колбасу на место. И предлагает другой своп из того же слота с другим вкусовым профилем
6. Показывает: «Понял. Пробуем другое в том же формате.»

**Постусловие:** taste_weight = −1.0 для куриного варёного, колбаса восстановлена

### UC-04: Пересборка списка вручную

**Актор:** Андрей
**Предусловие:** автосписок готов

**Основной поток:**
1. Открывает список, видит гречку
2. Думает «опять гречка»
3. Нажимает на позицию → детальная карточка
4. Нажимает «Заменить»
5. Видит 3 альтернативы из слота hot_meal/meal_base с похожей ценой
6. Выбирает булгур
7. Список обновляется, суммарный БЖУ пересчитывается

**Постусловие:** список скорректирован, сигнал «не хочу гречку» записан с весом −0.3

### UC-05: Planned Indulgence

**Актор:** Игорь (на сушке)
**Предусловие:** в профиле отмечено «кофе с выпечкой по пятницам»

**Основной поток:**
1. Пятница — список на день включает корректировку под выпечку
2. Игорь видит: «Сегодня кофе-брейк. Учтено в плане.»
3. Обед: куриная грудка + гречка — компенсация по калориям
4. Игорь не чувствует вины, не «срывается раз уж съел»
5. Система не комментирует пятничный круассан

**Постусловие:** план соблюдён, срыва нет, приверженность сохранена

### UC-06: Переход на этап 2 (три свопа)

**Актор:** Андрей, 3-я неделя
**Предусловие:** первый своп (куриное в нарезке) куплен дважды

**Основной поток:**
1. В воскресенье при генерации списка система показывает: «✅ Куриное в нарезке прижилось. Попробуем ещё пару улучшений?»
2. Предлагает 4 дополнительных свопа на выбор
3. Андрей выбирает 2 из 4
4. Список генерируется с тремя свопами, остальное — привычное

**Альтернативный поток (1a):** Андрей нажимает «Пока хватит» → система остаётся на этапе 1, предложит через 2 недели

### UC-07: Жёсткий бюджет (каждый рубль на счету)

**Актор:** Игорь (3000₽/нед, сушка, лактозоинтолерантность)

**Основной поток:**
1. Система видит: бюджет 3000₽, цель — сушка, белок 160г/день
2. Считает: 160 × 7 = 1120г белка в неделю. При текущем наборе стоимость 3480₽ — не влезает
3. Показывает: «Белковая цель на этот бюджет — сложно. Вот как оптимизировать корзину:»
4. Таблица: самые дешёвые источники белка в его магазинах (яйца С0, минтай мороженый, творог 5%, чечевица)
5. Итоговая корзина: 2870₽, ~140г белка/день
6. Система честно говорит: «До цели 20г/день — добавить ещё 2+ десятка яиц (+198₽) или скорректировать цель»
7. Игорь выбирает «скорректировать цель до 140г»

**Критерии приёмки:**
- Если бюджет не позволяет достичь цели — система говорит об этом прямо, не молча занижает
- Сортировка по ₽/г белка доступна как отдельный режим
- Промо и акции учитываются в расчёте

### UC-08: Комфортный бюджет

**Актор:** Алексей, 35 лет, 7000₽/нед, поддержание

**Основной поток:**
1. Список генерируется с приоритетом Taste Score + NutriScore A/B
2. Система видит: освоено 4900₽ из 7000₽
3. Показывает: «Есть 2100₽ запаса. Добавить?»
4. Варианты: рыба (лосось, дорадо), орехи, авокадо, ничего
5. Алексей добавляет лосось
6. В карточке лосося: Омега-3 2.3г/100г, NutriScore A. Пометка: [SR/MA] — ассоциация с кардиопротективным эффектом, не RCT на исходах

### UC-09: Премиум

**Актор:** Мария, 32 года, 12000₽/нед

**Основной поток:**
1. В профиле: «Цена не приоритет. Приоритет — состав.»
2. Value Score пересчитывается: w_nutriscore = 0.55, w_price = 0.05
3. Список содержит только NutriScore A/B товары
4. ВкусВилл присутствует как основной магазин
5. Система показывает сравнение: «За эту неделю корзина на 1400₽ дороже среднего — зато 18 из 20 позиций NutriScore A/B»

### UC-10: Тренер рекомендует приложение (invite-link)

**Актор:** Дарья (тренер), новый клиент Андрей

**Основной поток:**
1. Дарья генерирует invite-ссылку с параметрами: цель (набор), примерные КБЖУ-нормы, пометки (без лактозы), бюджет ~4500₽
2. Андрей устанавливает приложение, открывает ссылку
3. Онбординг предзаполнен данными от тренера: «Твой тренер уже настроил базовые параметры. Проверь и скорректируй.»
4. Андрей подтверждает/меняет под себя
5. Дарья получает уведомление «Клиент подключился»

**Критерии приёмки:**
- Invite-ссылка содержит только параметры, не личные данные
- Клиент видит, что именно предзаполнил тренер, и может это изменить
- Тренер НЕ видит детали корзины клиента без явного шеринга

### UC-11: Клиент делится недельным отчётом с тренером

**Актор:** Андрей (клиент), Дарья (тренер)
**Триггер:** конец недели, Андрей сам нажимает «Поделиться с тренером»

**Основной поток:**
1. Андрей видит воскресный отчёт: Белок 148г/день (цель 160) ⚠ −8%, Калории 2450/день (цель 2600) ⚠ −6%, Список выполнен на 80% (16/20), Не куплено: гречка (×2), минтай (×1)
2. Нажимает «Поделиться»
3. Дарья получает уведомление + краткую сводку

**Что отчёт НЕ содержит:**
- ❌ Конкретные цены покупок (приватность)
- ❌ Полный список товаров (только агрегаты)
- ❌ Данные о «запрещённых» продуктах (пахлава — только если пользователь сам включил)

**Бизнес-правило:** шеринг строго добровольный, каждую неделю отдельно. Пользователь может исключить из отчёта любые данные перед отправкой.

### UC-12: Дашборд тренера

3. Цветовая индикация: зелёный / жёлтый / красный по ключевым метрикам
4. Нажимает на клиента → видит агрегированный отчёт (только если клиент явно поделился)

**Критерии приёмки:**
- Дашборд только для клиентов, явно поделившихся данными
- Нет детальной информации — только агрегаты и этап прогрессии

### UC-13: Фото холодильника (стартовый инвентарь)

**Актор:** любой новый пользователь
**Триггер:** шаг онбординга «Что у тебя уже есть дома?»

**Основной поток:**

**Критерии приёмки:**

- Шаг полностью опциональный

## 6. User Stories

### Эпик 1: Онбординг

**US-01:** Как новый пользователь, я хочу рассказать о своих текущих привычках (что ем, где покупаю), чтобы система понимала мой контекст и не предлагала несовместимые замены.
- ✓ Онбординг ≤5 минут
- ✓ Вопросы о юзкейсах (завтрак, обед, ужин), не только о целях
- ✓ Можно пропустить любой шаг
- ✓ Нет вопросов о конкретных анализах или медицинских показателях

**US-02:** Как пользователь с ограничениями в питании (лактоза), я хочу указать непереносимость один раз, чтобы никогда не видеть такие продукты.
- ✓ Фильтр применяется ко всем системам (Quality Gate, CF, Value Score)
- ✓ При нулевом результате в слоте — явное сообщение, не пустой список
- ✓ Можно изменить в настройках в любой момент

### Эпик 2: Список и свопы

**US-03:** Как пользователь на наборе массы, я хочу получать список покупок каждое воскресенье автоматически, чтобы в будни не думать что купить.
- ✓ Список генерируется без действий пользователя
- ✓ Уведомление в воскресенье в удобное время (настраивается)
- ✓ Список открывается за ≤2 секунды
- ✓ Привычные продукты присутствуют

**US-04:** Как пользователь, который любит колбасу на бутерброд, я хочу получать замены только в том же формате (нарезка, 0 мин готовки), чтобы не видеть предложения купить сырую грудку.
- ✓ Замена имеет тот же convenience_tier
- ✓ Замена имеет хотя бы один совпадающий use_context
- ✓ В карточке замены показано «тот же формат использования»
- ✓ Нет замен между слотами без явного запроса

**US-05:** Как пользователь, я хочу свайпом отклонить товар и сразу увидеть альтернативу, чтобы коррекция занимала ≤3 тапа.
- ✓ Свайп влево → шторка с 3 альтернативами
- ✓ Каждая альтернатива: название, цена, главное отличие одной строкой
- ✓ Тап → товар заменён, список обновлён
- ✓ Кнопка «Оставить оригинал» всегда видна

### Эпик 3: Прогрессивное погружение

**US-06:** Как пользователь на этапе 1, я хочу сам решать когда переходить к большему числу свопов, чтобы не чувствовать давления.
- ✓ Система предлагает переход, не навязывает
- ✓ «Пока хватит» — всегда доступная кнопка
- ✓ Темп (медленно/комфортно/быстро) меняется в настройках
- ✓ Откат без объяснений, без уведомлений

**US-07:** Как пользователь на этапе 4 (автопилот), я хочу чтобы мои любимые продукты (кофе, бородинский) никогда не исчезали из списка.
- ✓ «Неприкосновенные» продукты всегда в списке
- ✓ Система не предлагает их заменить, даже если NutriScore низкий
- ✓ Отмечены иконкой «моё постоянное»
- ✓ Можно убрать из «неприкосновенных» в любой момент

### Эпик 4: Честность и прозрачность

**US-08:** Как пользователь, я хочу понимать почему товар получил такую оценку, чтобы доверять системе.
- ✓ Каждый критерий раскрывается в детальной карточке
- ✓ Уровень уверенности ([RCT] / [SR/MA] / [i]) виден
- ✓ Нет утверждений типа «доказано снижает риск» без RCT-источника
- ✓ Дисклеймер «не является медицинской рекомендацией» на нутр. экранах

**US-09:** Как пользователь, я хочу видеть сколько реально стоит белок в разных продуктах одного слота.
- ✓ ₽/г белка для каждого товара в слоте
- ✓ Медиана по слоту видна («дешевле среднего» / «дороже среднего»)
- ✓ Явная пометка «Калькулятор»
- ✓ Работает только для товаров с доступными нутрифактами

### Эпик 5: Taste Engine

**US-10:** Как пользователь через 6+ недель использования, я хочу чтобы предложения становились точнее и учитывали мои реальные вкусы.
- ✓ После 50+ взаимодействий включён CF
- ✓ Отклонённые товары не предлагаются повторно в том же слоте
- ✓ Повторно купленные остаются в списке по умолчанию
- ✓ Можно сбросить вкусовой профиль в настройках

- ✓ Этап 1 (один своп) работает полностью
- ✓ Пользователь может отклонить своп без последствий
- ✓ Темп настраивается

## 10. Доказательная база (Summary)

| Функция | Обоснование | Уровень |
|---------|------------|---------|
| Quality Gate: NutriScore-2023 как основа | RCT, n=21 159, OR=29.0 | ✅ RCT |
| NOVA-классификация | Umbrella review, 45 SR/MA, n=9.8M | ⚠ SR/MA (не RCT на исходах) |
| Ценовая видимость меняет выбор | SR/MA 34 исследования, 16 RCT, Lancet 2024 | ✅ RCT |
| Готовый список снижает decision fatigue | Narrative review, PМС 2025 | ℹ Review |
| Behaviour substitution (своп) | SR по BCT effectiveness | ⚠ SR |
| Постепенность > резкая смена | RCT по постепенному снижению сахара | ✅ RCT |
| Habit formation ≈ 59 дней | SR/MA, 27 исследований, PМС 2024 | ⚠ SR/MA |
| Planned indulgence | SR по food choice barriers | ⚠ SR |
| Персонализация > генерические советы | Review of RCTs, Food & Function 2026 | ✅ RCT |
| Feedback + мониторинг = сильнейший BCT | RS, 12+ RCT мобильных приложений | ✅ RCT |

## 11. Риски и митигации

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Магазины закроют парсинг | Высокая | Договориться об официальном API, либо партнёрство |
| Неполнота базы Честного знака | Средняя | Fallback на OpenFoodFacts + ручной ввод |
| Пользователь не доверяет автосписку | Средняя | Объяснение Score с деталями, уровни уверенности, история изменений |
| Актуальность цен отстаёт | Низкая | TTL кэша ≤24 ч, индикатор «цена актуальна на ...» |
| Парсинг внутренних API мобильных приложений магазинов | Высокая | Реверс REST API мобильных приложений → долгосрочно: официальные партнёрства |
| Пользователь бросает после первой недели | Средняя | Прогрессивная модель (1 своп, не ребилд), Taste Engine для retention |

## 12. Фоновые триггеры (без действий пользователя)

| Триггер | Действие системы | Уведомление |
|---------|-----------------|-------------|
| Пятница вечер | Пересчёт списка на следующую неделю | «Список на неделю готов» |
| Цена на товар упала >15% | Обновление в списке | «Куриное филе в Пятёрочке −22%» |
| Новый товар прошёл Quality Gate | Добавляется в Taste Engine | «Новинка — похожие тренеры уже берут» |
| Пользователь 10 раз купил один товар | Повышение веса в taste_vector | Нет уведомления (тихий сигнал) |
| Цена на товар выросла >20% | Поиск альтернативы | «⚠ Творог подорожал, нашли дешевле» |
| Промо/акция на товар из категории пользователя | Проверка через Quality Gate → предложение | «Минтай по акции — дешевле на 30%» |
