# BUFF EAT — Business Requirements Document

**Версия:** 2.0 | **Дата:** Март 2026 | **Статус:** Draft (post-critical-review)

---

## 1. Контекст и проблема

### 1.1 Целевой пользователь

Фитнес-тренер или спортсмен, который:

- Знает свои макро/микро-нормы, но не успевает переносить их в конкретные товары
- Не хочет ходить по магазинам ради сравнения цен
- Не может быстро понять: «этот творог за 189₽ — нормально или развод?»

**Ключевой инсайт:** проблема не в сканировании — проблема в том, что пользователь каждый раз делает одну и ту же аналитическую работу заново, вручную, в голове.

### 1.2 Персонажи

**Персонаж A: Андрей**, 29 лет, менеджер (клиент тренера)

| Параметр | Значение |
|----------|----------|
| Цель | Набрать 5 кг мышц |
| Бюджет | 4500₽/нед на еду |
| Магазины | Пятёрочка (рядом), Перекрёсток (по пути) |
| Готовит | 20–30 мин вечером, по выходным нормально |
| Завтрак | Бутерброд с колбасой + кофе |
| Болит | Не добирает белок, колбасу любит и не готов отказываться |

**Персонаж B: Игорь**, 29 лет, на сушке

| Параметр | Значение |
|----------|----------|
| Цель | Сбросить 8 кг жира, сохранить мышцы |
| Бюджет | 3000₽/нед (жёсткий) |
| Магазины | Пятёрочка, Магнит |
| Готовит | Минимально, некогда |
| Болит | Дорого, лень, вечером ест пахлаву горстями |
| Ограничения | Непереносимость лактозы |

**Персонаж C: Дарья**, 34 года, фитнес-тренер

| Параметр | Значение |
|----------|----------|
| Роль | B2B-пользователь (в v2) |
| Задача | Контролировать продуктовую корзину клиентов |
| Болит | Клиенты не выполняют план питания → нет прогресса → уходят |

### 1.3 Карта реальных болей

| # | Боль | Корневая причина | Как сейчас решают |
|---|------|-----------------|-------------------|
| P1 | Физически тяжело добирать белок | Однообразие, нет замен в привычном контексте | Протеиновые порошки — дорого |
| P2 | Дорого питаться правильно | Нет инструмента сравнения ₽/нутриент | Покупают «дешевле» = хуже состав |
| P3 | Вечером лень, думать → срыв | Decision fatigue → к концу дня нет когнитивного ресурса | Волевое усилие — не работает долго |
| P4 | Непонятно что купить вместо привычного | Замены предлагают без учёта юзкейса | Гуглят, находят несовместимые советы |
| P5 | Не могу отказаться от круассана/пахлавы | Cravings ≠ слабоволие, это нейробиология привычки | Диета → срыв → отказ от изменений |
| P6 | Не понимаю, стоит ли товар своих денег | Нет инструмента оценки value | Сравнивают цену за 100г — неверная метрика |

### 1.4 Чего продукт НЕ пытается решить

- Заменить нутрициолога или врача
- Лечить расстройства пищевого поведения
- Строить точные медицинские рекомендации без анализов
- Полностью менять пищевые привычки за неделю

## 2. Продуктовое видение

### 2.1 Формулировка

**BUFF EAT** убирает необходимость думать о еде каждый день — за счёт готового списка покупок, который подстраивается под привычки пользователя и тихо улучшает их.

### 2.2 Ценностное предложение

- **Для клиента тренера:** «Ты продолжаешь есть привычное — только лучшие версии того же самого»
- **Для тренера:** «Твои клиенты приходят с реальной корзиной, которая соответствует плану питания»

### 2.3 Принципы продукта

1. **Постепенность** — 3–5 свопов в неделю, не ребилд корзины
2. **Юзкейс первичен** — замена только внутри одного функционального слота
3. **Честность** — показываем уровень уверенности у каждого утверждения
4. **Автопилот** — система работает без участия пользователя, он только корректирует
5. **Без запретов** — любимые продукты неприкосновенны
6. **Гибкость** — плавное погружение, пользователь контролирует темп

## 3. Функциональные требования (MVP)

### RF-01: Онбординг

**Цель:** за ≤5 минут получить достаточно данных для первого набора свопов. Не перегружать.

**Шаги:**

**Шаг 1 — «Что тебя достало?»** (1 экран, чекбоксы)
- Варианты: не добираю белок / дорого / лень / срываюсь / не знаю что купить

**Шаг 2 — Текущие привычки по слотам** (2–3 экрана)
- Завтрак: что обычно? (бутерброд / яичница / каша / пропускаю)
- На бутерброд: колбаса / сыр / паштет / рыба
- Ужин: готовишь сам / полуфабрикаты / заказываешь

**Шаг 3 — Контекст** (1 экран)
- Цель: набор / сушка / поддержание
- Бюджет: слайдер
- Магазины: мультиселект

**Шаг 4 — Ограничения** (1 экран, опционально)
- Аллергены, нелюбимые продукты: «это никогда не трогай»

**Бизнес-правило:** система НЕ задаёт вопрос о нутритивных дефицитах без анализов. Использует популяционные данные по дефицитам для РФ (D, Mg, йод) с явным дисклеймером.

### RF-02: Quality Gate

**Цель:** отфильтровать товары до того, как они попадут к пользователю.

**Инструмент:** NutriScore-2023. Используется несмотря на отсутствие официального принятия в России — российская аудитория склонна доверять международным системам оценки питания выше, чем локальным государственным (ADR-008).

**Алгоритм** (только доказательно обоснованные критерии):

**TIER-1 — Блокировка** (`gate_passed = false`):
- NutriScore D или E (алгоритм NutriScore-2023) [RCT]
- NOVA класс 4 И NutriScore C или ниже — одновременно [SR/MA]
- Промышленные трансжиры > порога EFSA [RCT + механистические данные]

**TIER-2 — Предупреждение** (`gate_passed = true, warning = true`):
- NOVA класс 4 при NutriScore B/C
- Эмульгаторы E466, E471–E475 в составе [RCT, короткие]
- Натрий > 1.5 г/100г

**TIER-3 — Информация** (без штрафа):
- Нитрит натрия (E250) — отображается как факт с контекстом [когорты, нет RCT]
- Прочие аддитивы — показываем без оценки

**Что система НЕ делает:**
- Не штрафует E102/E110/E124 у взрослых (данные только на детях с СДВГ)
- Не использует позицию ингредиента в составе как критерий
- Не оценивает % мяса как напрямую нутритивный показатель

**Бизнес-правило:** каждый критерий отображается с уровнем уверенности:
- ✅ `[RCT]` — подтверждено рандомизированными исследованиями
- ⚠ `[SR/MA]` — подтверждено систематическими обзорами / мета-анализами
- ℹ `[i]` — обсервационные данные, не RCT

### RF-03: Value Score + математическая модель

**Цель:** показать экономическую эффективность товара под конкретный профиль и оптимизировать корзину.

**Два явно разделённых блока:**

**Блок A — Нутри-оценка** `[RCT]`:
- NutriScore-2023 (алгоритм A–E) — валидирован в RCT

**Блок B — Калькулятор** (арифметика, не доказательный инструмент):
- ₽ за 1г белка
- ₽ за 100 ккал
- ₽ за единицу приоритетного нутриента из профиля
- Сравнение с медианой по слоту (cheaper / average / expensive)

**Бизнес-правило:** Блок B отображается с плашкой: «Калькулятор — полезный ориентир, не медицинская рекомендация.»

#### Математическая модель генерации списка (LP-оптимизация)

Генерация недельного списка — задача линейного программирования:

```
Максимизировать:
  Σ (value_score_i × x_i)

При ограничениях:
  Σ (price_i × x_i) ≤ weekly_budget
  Σ (protein_i × x_i) ≥ target_protein (г/нед)
  Σ (calories_i × x_i) ≥ target_calories_min
  Σ (calories_i × x_i) ≤ target_calories_max
  sacred_items ⊆ x (неприкосновенные всегда в списке)
  x_i ∈ {0, 1, 2, ...}  (количество единиц)
```

**Реализация:** `scipy.optimize.linprog` или `PuLP`. Холодный старт: задача решается на эталонной базе `universal_products`.

#### Ранжирование свопов (Composite Score)

Внутри слота (convenience_tier + use_context) товары ранжируются по:

```
composite_score = w_nutriscore × norm(nutriscore_points)
               + w_price     × norm(1 / price_per_protein_g)
               + w_deficit   × norm(deficit_nutrient_amount)
```

Веса калибруются по бюджету пользователя:

| Бюджет | w_nutriscore | w_price | w_deficit |
|--------|-------------|---------|-----------|
| <3000₽ | 0.25 | 0.55 | 0.20 |
| 3000–6000₽ | 0.35 | 0.35 | 0.30 |
| >6000₽ | 0.45 | 0.15 | 0.40 |

**Персонализация приоритетного нутриента по цели:**

| Цель | Приоритетный нутриент |
|------|----------------------|
| Набор | Белок, калории |
| Сушка | Белок, клетчатка |
| Поддержание | Белок, NutriScore |

### RF-04: Своп-система (ядро продукта)

**Цель:** предлагать 3–5 замен в неделю строго внутри функционального слота.

**Таксономия юзкейсов:**

Каждый товар в базе получает два атрибута:

**convenience_tier** (INTEGER):
- 1 = ready-to-eat (открыл — ешь, 0 мин)
- 2 = quick-cook (5–10 мин, минимум действий)
- 3 = meal-base (20+ мин)

**use_context** (TEXT[]):
- `sandwich` — на хлеб, без нагрева
- `fry_with_egg` — жарить с яйцом
- `hot_meal` — горячее блюдо
- `cold_snack` — холодный перекус
- `snack` — перекус, носимый
- `breakfast_ready` — утро, быстро
- `pre_workout` — до тренировки
- `post_workout` — после тренировки
- `topping` — добавка к чему-то

**Алгоритм подбора замены:**

1. Определить `convenience_tier` текущего товара
2. Определить `use_context` текущего товара
3. Найти товары с совпадающим tier И хотя бы одним совпадающим context
4. Из них: прошедшие Quality Gate
5. Из них: отсортировать по composite_score под профиль
6. Вернуть топ-3:
   - #1 → лучший по NutriScore в слоте
   - #2 → лучший ₽/целевой нутриент в слоте
   - #3 → следующий лучший composite_score в слоте

**Бизнес-правило:** система НИКОГДА не предлагает своп между разными `convenience_tier` без явного запроса пользователя.

**Пример:**

Пользователь берёт: Колбаса Докторская Черкизово
- Слот: ready_to_eat + [sandwich, fry_with_egg]

❌ Система НЕ предлагает:
- Куриную грудку 1 кг (сырую) — другой слот (meal_base)

✅ Система предлагает:
- Колбаса куриная «Петруха» — тот же слот, лучше состав
- Нарезка индейки ВкусВилл — тот же слот, чище
- Куриное филе варёное в нарезке (вакуум) — тот же слот, +8 г белка/100г

### RF-05: Прогрессивная модель погружения

**4 этапа с триггерами перехода:**

| Этап | Название | Что предлагается | Триггер перехода |
|------|----------|------------------|------------------|
| 1 | Первые свопы | 3–5 замен на выбор, пользователь выбирает сколько взять | Хотя бы 2 свопа куплены 2+ раза |
| 2 | Расширение | До 5–7 замен параллельно | 3+ свопов прижились |
| 3 | Полкорзины | Автогенерация ~50% списка | Пользователь нажал «хочу больше» |
| 4 | Автопилот | Полный список + 20–30% неприкосновенных | Пользователь включил сам |

**Бизнес-правила:**
- Откат на предыдущий этап — в любой момент, без объяснений
- Темп (медленно / комфортно / быстро) настраивается в любой момент
- «Неприкосновенные» товары — система никогда не свапает и не убирает из списка
- Нет уведомлений с давлением («ты не выполнил», «осталось 3 дня»)
- Нет стриков, бейджей, «ты молодец»

**Тон сообщений:**

| Этап | Тон | Пример |
|------|-----|--------|
| 1 | Нейтральный, без обязательств | «Вот 3 варианта — попробуешь какой-нибудь?» |
| 2 | Лёгкое поощрение факта | «Куриная грудка уже третью неделю в списке» |
| 3 | Предложение расширить | «Хочешь, возьмём больше на себя?» |
| 4 | Партнёрский | «Список готов. Твои любимые продукты на месте.» |

### RF-06: Список покупок

**Генерация:** каждое воскресенье автоматически + по запросу.

**Структура списка:**
- **Шапка:** Период / Итого ₽ / Прогресс по белку и калориям
- **Секции** по категориям товаров
- **Карточка товара:** Название · Магазин · Цена · NutriScore · ₽/г белка (если релевантно) / Тег: «Автоподобрано» / «Моё постоянное»
- **Свопы недели** — выделенный блок (3–5 предложений)
- **Футер:** Суммарно Белок Хг / Жиры Хг / Углеводы Хг / ⚠ Дефицит (если есть)

**Коррекция:**
- Свайп влево → замены в том же слоте (топ-3)
- Тап → детальная карточка товара
- «Другой магазин» → аналог в другом магазине из профиля

### RF-07: Planned Indulgence

**Цель:** не запрещать любимые продукты, а встраивать их в план.

**Механика:**
- Пользователь добавляет «индульгенции» в профиле: кофе, круассан, пахлава
- Хранятся как: `product_id + serving_g + day_of_week + meal_context`
- Система учитывает их калорийность и БЖУ при генерации списка
- **Компенсация недельная, не дневная** — если пятничная выпечка > остатка дня, компенсация перераспределяется по неделе
- Если индульгенция превышает суточный остаток — не отказ, а предложение перераспределить по неделе
- Никаких «ты съел лишнее» — только тихая компенсация

**UI-правило:** тон исключительно нейтральный. «Сегодня кофе-брейк. Учтено в плане.» Никаких оценочных суждений.

**Бизнес-правило:** система НИКОГДА не показывает негативную оценку «индульгентных» продуктов. Только нейтральный факт, если пользователь сам спросит.

### RF-08: Дисклеймеры и уровни уверенности

Обязательно присутствуют на каждом экране с нутритивными утверждениями.

**В карточке товара:**
- ℹ NutriScore основан на алгоритме NutriScore-2023, валидирован [RCT]
- ℹ Оценка состава по NOVA и данным EFSA [SR/MA]
- ₽/г белка — калькулятор, не медицинская рекомендация

**В онбординге:**
- «Рекомендации по нутриентам основаны на популяционных данных ВОЗ/Роспотребнадзора. Для персональных рекомендаций — сдайте анализы.»

### RF-10: Error Flows

**Цель:** система ведёт себя предсказуемо в граничных ситуациях.

| Ситуация | Действие системы |
|----------|-----------------|
| Бюджет не покрывает белковую цель | Честно сообщаем: «На X₽ в неделю ваша белковая цель (Xг) недостижима. Вот как оптимизировать:» + таблица ₽/г белка |
| Слот пустой после Quality Gate | «В этом формате нет подходящих товаров в вашем магазине. Добавить из другого магазина?» |
| Нет цен для магазина пользователя | Показываем товары с пометкой «Цена не актуальна» и последней известной ценой |
| Данные о составе отсутствуют | Товар не участвует в подборе свопов. В карточке: «Состав не верифицирован» |
| Парсер временно недоступен | Список генерируется из кэша. «Цены актуальны на [дата]» |
| Все любимые товары заблокированы QG | Ни один «неприкосновенный» товар не блокируется — только предупреждение в карточке |
| Интернет недоступен | Последний список доступен офлайн с плашкой «Данные актуальны на [дата]» (как в 2ГИС) |

### RF-11: Политика данных

**Что собираем:**
- Профиль питания (цель, бюджет, магазины, ограничения)
- Взаимодействия с товарами (add/remove/buy/dislike)
- Planned Indulgence (категории, не конкретные товары если не нужно)

**Что НЕ собираем:**
- Данные о реальных покупках (нет интеграции с банками/кассами)
- Медицинские показатели (анализы, биомаркеры)
- Геолокацию в реальном времени

**Хранение и удаление:**
- Профиль: по запросу пользователя — полное удаление в 72ч
- Взаимодействия: анонимизируются через 180 дней
- Вкусовые сигналы (taste_weight): привязаны к user_id, удаляются вместе с аккаунтом

**Прозрачность:**
- Пользователь видит в настройках что именно хранится
- «Что система знает обо мне» — отдельный раздел

---

### RF-12: Двухуровневая стратегия данных

**Tier A — Универсальные продукты** (`universal_products`):
- Яйца, куриное филе/окорочка, творог, гречка, овсянка, рис, минтай, чечевица, капуста, морковь и аналоги
- Полный верифицированный состав (вручную + OpenFoodFacts)
- Не имеют дубликатов — один эталонный запис на «куриное филе»
- Доступны с первого дня без парсинга
- ~50–80 товаров на старте

**Tier B — Брендовые товары** (`products`):
- Конкретные SKU из магазинов: «Куриная колбаса Петруха 300г», «Йогурт Чудо клубника 500г»
- Получаются парсингом каталогов магазинов + фильтрация через Quality Gate
- Дополняют эталонную базу
- Только если прошли Quality Gate

**Офлайн-режим:**
- Каждый список имеет метку «Цены актуальны на [дата]»
- Предупреждение показывается если данные старше 48 часов
- Последний список всегда доступен без интернета

## 5. Use Cases

### UC-01: Первый визит (нет аккаунта)

**Актор:** Андрей (Персонаж A)

**Основной поток:**
1. Видит экран «Что тебя достало в питании?» — выбирает «не добираю белок» и «не знаю что покупать»
2. Отвечает на вопросы о завтраке: бутерброд с колбасой
3. Указывает цель (набор), бюджет (4500₽), магазины (Пятёрочка, Перекрёсток)
4. Система предлагает шаблон «Качок на наборе» как стартовую корзину
5. Видит 3–5 свопов: «Докторская → Куриное варёное в нарезке», «Белый хлеб → Цельнозерновой» и др.
6. По каждому свопу: то же использование, разница в белке, разница в цене
7. Выбирает 2 свопа, нажимает «Добавить в список»

---

### UC-02: Вечерний поход в Пятёрочку

**Актор:** Игорь (Персонаж B), после работы, усталый

**Основной поток:**
1. Открывает приложение у входа в магазин
2. Видит список — 8 позиций, отфильтрованы под Пятёрочку
3. На позиции «Творог» видит: ⚠ «Закончился — вот замена в том же слоте»
4. Система предлагает 2 аналога: тот же cold_snack, тот же convenience_tier
5. Выбирает первый, ставит галочку

**Альтернативный поток (4a):** товар есть, но цена выросла на 25%+ → «Подорожал. Вот аналог дешевле.»

---

### UC-03: Своп не подошёл

**Актор:** Андрей

**Основной поток:**
1. Андрей купил «Куриное в нарезке» на первой неделе
2. На второй неделе свайпает влево → «Не моё»
3. Видит: «Что не так?» — дорого / вкус / упаковка / просто не хочу
4. Выбирает «Вкус не понравился»
5. Система возвращает колбасу и предлагает другой своп из того же слота

---

### UC-04: Жёсткий бюджет

**Актор:** Игорь (3000₽/нед, сушка, лактозоинтолерантность)

**Основной поток:**
1. Система считает: 160г белка × 7 дней = 1120г/нед. При текущей корзине стоимость 3480₽ — не влезает
2. Показывает: «Белковая цель на этот бюджет — сложно. Вот как оптимизировать:»
3. Таблица: самые дешёвые источники белка (яйца, минтай, чечевица, творог 0%)
4. Итоговая корзина: 2870₽, ~140г белка/день
5. Система честно: «До цели 20г/день — добавить ещё 2 десятка яиц (+140₽) или скорректировать цель»
6. Игорь выбирает «скорректировать цель до 140г»

### UC-05: Planned Indulgence

**Актор:** Игорь (на сушке)

**Основной поток:**
1. В профиле: «кофе с выпечкой по пятницам» — 350 ккал, meal_context=cafe_break
2. Список на неделю генерируется с учётом: остальные дни чуть строже по калориям
3. Пятница: «Кофе-брейк. Учтено в плане.» — без оценок
4. Обед в пятницу: немного больше белка, меньше жиров — тихая компенсация
5. Игорь не чувствует вины, план соблюдён

### UC-06: Тренер рекомендует приложение

**Актор:** Дарья (тренер), новый клиент Андрей

**Основной поток:**

2. Андрей открывает — онбординг предзаполнен: «Твой тренер настроил базовые параметры»
3. Андрей подтверждает/меняет под себя
4. Дарья получает уведомление «Клиент подключился»

## 6. User Stories

### Эпик 1: Онбординг

**US-01:** Как новый пользователь, я хочу рассказать о своих привычках, чтобы система понимала мой контекст.
- ✓ Онбординг ≤5 минут
- ✓ Вопросы о юзкейсах (завтрак, обед, ужин), не только о целях
- ✓ Стартовая корзина предложена из 5 шаблонов
- ✓ Можно пропустить любой шаг

**US-02:** Как пользователь с ограничениями (лактоза), я хочу указать непереносимость один раз.
- ✓ Фильтр применяется ко всем системам
- ✓ При нулевом результате в слоте — явное сообщение, не пустой список

### Эпик 2: Список и свопы

**US-03:** Как пользователь на наборе, я хочу список покупок каждое воскресенье автоматически.
- ✓ Генерируется без действий пользователя
- ✓ Список открывается за ≤2 секунды

**US-04:** Как пользователь, который любит колбасу на бутерброд, я хочу замены только в том же формате.
- ✓ Замена: тот же convenience_tier + хотя бы один use_context

**US-05:** Как пользователь, я хочу 3–5 свопов на выбор, чтобы самому решать сколько брать.
- ✓ 3–5 предложений за неделю, пользователь выбирает
- ✓ «Не сейчас» — всегда доступно

### Эпик 3: Прогрессивное погружение

**US-06:** Как пользователь на этапе 1, я хочу сам решать когда переходить к большему числу свопов.
- ✓ Система предлагает переход, не навязывает
- ✓ «Пока хватит» — всегда доступная кнопка

**US-07:** Как пользователь на этапе 4, я хочу чтобы мои любимые продукты никогда не исчезали.
- ✓ «Неприкосновенные» всегда в списке

### Эпик 4: Честность

**US-08:** Как пользователь, я хочу понимать почему товар получил такую оценку.
- ✓ Уровень уверенности ([RCT] / [SR/MA] / [i]) виден
- ✓ Нет утверждений «доказано снижает риск» без RCT-источника

## 9. Критерии готовности MVP

**Онбординг:**
- ✓ Пользователь проходит за ≤5 минут
- ✓ Стартовая корзина из шаблона предложена немедленно

**Список:**
- ✓ Генерируется за ≤30 секунд (LP-оптимизация на universal_products)
- ✓ Содержит только товары, прошедшие Quality Gate
- ✓ Каждый товар имеет NutriScore и/или ₽/г белка

**Свопы:**
- ✓ 3–5 предложений в неделю
- ✓ Замены только внутри одного use_context и convenience_tier
- ✓ Коррекция одной позиции ≤3 тапа

**Данные:**
- ✓ universal_products заполнен (50+ эталонных позиций)
- ✓ Цены из хотя бы 2 магазинов (Пятёрочка + Перекрёсток)

**Прозрачность:**
- ✓ Уровень уверенности у каждого нутритивного утверждения
- ✓ Дисклеймер на всех нутритивных экранах
- ✓ Error flows покрыты

## 11. Риски и митигации

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Магазины закроют парсинг | Высокая | Реверс мобильных API → долгосрочно: официальные партнёрства |
| Неполнота базы (Tier B) | Средняя | Tier A (universal_products) достаточен для MVP; Tier B расширяется постепенно |
| NutriScore не знаком россиянам | Средняя | Объяснение при первом показе; акцент на «международная система» |
| Актуальность цен отстаёт | Средняя | TTL кэша ≤24ч, индикатор «цена актуальна на...», офлайн с датой |
| Пользователь бросает после первой недели | Средняя | Прогрессивная модель (3–5 свопов на выбор, не ребилд); bootstrap-корзины снижают трение онбординга |
| LP не решается при жёстком бюджете | Низкая | Fallback: ослабить ограничение по белку + честное сообщение |

| Триггер | Действие системы | Уведомление |
|---------|-----------------|-------------|
| Воскресенье вечер | Генерация списка на неделю | «Список на неделю готов» |
| Цена на товар упала >15% | Обновление в списке | «Куриное филе в Пятёрочке −22%» |
| Товар прошёл Quality Gate (новый) | Добавляется в Tier B | (нет уведомления) |
| Цена на товар выросла >20% | Поиск альтернативы в слоте | «⚠ Творог подорожал, нашли дешевле» |

| Данные о ценах старше 48 часов | Плашка на списке «Данные актуальны на...» | (нет push-уведомления) |
