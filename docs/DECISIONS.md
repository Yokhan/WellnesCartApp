## ADR-001: NutriScore-2023 как основа Quality Gate

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** нужна система оценки качества продуктов для автоматической фильтрации.

**Решение:** алгоритм NutriScore-2023.

**Обоснование:** единственная система маркировки с серьёзной RCT-валидацией (n=21 159, OR=29.0). Второй независимый RCT (n>5000) подтвердил эффект.

**Последствия:**
- Нужен точный расчёт по алгоритму 2023 года (не 2017)
- NOVA — дополнительный критерий (SR/MA, не RCT)
- Собственные «штрафные формулы» за E-шники убраны

## ADR-002: Юзкейс-первичные свопы (convenience_tier + use_context)

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** система предлагала замены по нутриентам (колбаса → куриная грудка сырая). Пользователь покупает колбасу не ради белка, а чтобы сделать бутерброд за 0 минут.

**Решение:** каждый товар получает `convenience_tier` (1/2/3) и `use_context` (TEXT[]). Свопы — только внутри одного слота.

**Последствия:**
- Каждый товар в базе нужно разметить (LLM-assisted → ручная корректировка)
- SQL-запросы подбора замен включают обязательный фильтр по tier и context

## ADR-003: Прогрессивная модель 4 этапов

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** первая версия предполагала сразу генерировать полную «правильную» корзину.

**Решение:** 4-этапная прогрессивная модель с пользовательским контролем темпа.

**Обоснование:**
- SR/MA (PMC 2024): медиана формирования привычки = 59 дней, только 23% достигают автоматизма
- RCT: инкрементальные изменения по ~10% не снижают удовольствие

**Последствия:**
- progression_stage (1-4) в user_nutrition_profiles
- Переходы только по явным триггерам
- Никакого принудительного расписания

## ADR-005: CF (Taste Engine) — retention-инструмент, не нутритивная персонализация

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** коллаборативная фильтрация — мощный инструмент. В контексте питания нужно честно определить её роль.

**Решение:** CF позиционируется как инструмент retention, НЕ как медицинская персонализация.

**Последствия:**
- CF не влияет на Quality Gate
- Taste Score НЕ смешивается с NutriScore в один score
- CF реализуется в v1.5, не MVP

## ADR-007: Двухуровневая стратегия данных (Tier A + Tier B)

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** база данных не может быть полной на старте. Нужен способ запустить MVP без полного парсинга магазинов.

**Проблема с одноуровневой стратегией:**
- Парсинг всех магазинов нестабилен (магазины блокируют)
- Состав брендовых товаров сложно верифицировать
- Риск попасть в «мусорный» продукт без данных о составе

**Решение:** две таблицы с разной природой данных:
- **Tier A (`universal_products`):** ~50–80 эталонных позиций, верифицированный состав, без дубликатов. Достаточен для LP-оптимизации на старте.
- **Tier B (`products`):** брендовые SKU из парсинга + авто-QG. Расширяется по мере работы.

**Последствия:**
- Холодный старт MVP не требует парсинга
- LP на Tier A работает с первого дня
- Tier B добавляет разнообразие (брендовые замены внутри слотов)
- При ссылке из списка на Tier A показывается «универсальный продукт, цена зависит от бренда»

## ADR-008: NutriScore в России — международное доверие vs локальные альтернативы

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** NutriScore не принят официально в России. Роспотребнадзор продвигает «светофорную» маркировку.

**Варианты:**
1. Российский «Светофор» Роспотребнадзора
2. NutriScore-2023
3. Собственная формула

**Решение:** NutriScore-2023, несмотря на отсутствие официального статуса в РФ.

**Обоснование:**
- Российская wellness-аудитория доверяет международным системам выше, чем государственным
- NutriScore имеет RCT-валидацию. Российский «Светофор» — нет
- Российская wellness-наука (в части пищевых рекомендаций) преимущественно обсервационна или рекламна

**Митигация рисков:**
- Дисклеймер: «NutriScore — международная система, в России без официального статуса»
- Не утверждаем что это «государственная рекомендация»

**Последствия:**
- Продукты оцениваются по алгоритму NutriScore-2023
- UI объясняет что такое NutriScore при первом показе

## ADR-009: 3–5 свопов в неделю (не 1)

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** исходная спецификация предполагала «1 своп в неделю».

**Проблема:** 1 своп/нед = 52 свопа/год = крайне медленная прогрессия. Для MVP с Claude Code агентами сроки сжаты, нужно давать пользователю больше ценности быстрее.

**Решение:** 3–5 свопов в неделю на выбор пользователя. Пользователь сам решает сколько взять.

**Обоснование:**
- «3–5 рекомендаций в неделю» — достаточно, не перегружает
- User choice: предлагаем 3–5, пользователь выбирает 1–5 — нет давления
- Прогрессивная модель этапа 1 = выбор из предложенных свопов, а не принудительный 1 своп

**Последствия:**
- RF-04 обновлён: «3–5 свопов на выбор»
- List Builder генерирует 3–5 кандидатов по composite_score
- Триггер перехода на этап 2: хотя бы 2 свопа куплены 2+ раз

## ADR-010: Taste Engine — v1.5, не MVP

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** исходная спецификация включала SVD + pgvector + кластеризацию в MVP.

**Проблема:**
- Taste Engine требует: pgvector setup, SVD-алгоритм, кластеризацию пользователей
- На холодном старте (0 пользователей) CF бесполезна — нет данных
- Over-engineering для MVP

**Решение:** MVP работает на Quality Gate + Value Score (composite_score). Taste Engine реализуется в v1.5.

**Что собираем уже в MVP:** user_product_interactions и user_product_taste — сигналы копятся, но не используются для ранжирования.

**Последствия:**
- user_taste_vectors, user_clusters — не создаются в MVP (закомментированы в schema)
- Swap ranking в MVP: по composite_score, не Taste Score
- Свайп «Не моё» → обновляет taste_weight → будет использовано в v1.5

## ADR-011: Vision AI — v1.5, не MVP

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** исходная спецификация включала «фото холодильника» в MVP онбординг.

**Проблема:**
- Fine-tuned YOLOv8 на российских продуктах — отдельный ML-проект
- LLM Vision на российских продуктах: точность неизвестна
- Не критично для core value (генерация списка работает без фото)

**Решение:** Vision AI откладывается в v1.5. В MVP — ручной ввод через поиск.

**Последствия:**
- Онбординг: «Что у тебя уже есть?» — поиск по названию, не фото
- RF-10 убран из MVP scope, задокументирован в Out of Scope

## ADR-012: Bootstrap-корзины как стартовая стратегия

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** LP-оптимизация на холодном старте требует базы данных. Без bootstrap-корзин пользователь видит пустой список.

**Решение:** 5 эталонных корзин покрывают типичных российских покупателей × ценовые сегменты.

**Корзины:**
1. «Качок на наборе» (~4500₽/нед, 185г белка/день)
2. «Сушка бюджетная» (~2800₽/нед, 140г белка/день)
3. «Здоровое питание офисника» (~3800₽/нед, 115г белка/день)
4. «Премиум ЗОЖ» (~8500₽/нед, 135г белка/день)
5. «Студент-спортсмен» (~2000₽/нед, 120г белка/день)

**Последствия:**
- basket_templates + basket_template_items в схеме
- Онбординг: пользователь выбирает ближайший шаблон → LP корректирует под конкретный профиль
- Шаблоны хранятся в Tier A (universal_products)

## ADR-013: Planned Indulgence — недельная компенсация, не дневная

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** исходная спецификация: компенсация в «других приёмах того же дня».

**Проблема:**
- Дневная компенсация жёсткая: если вечером пахлава — весь день под вопросом
- Психологически давит: пользователь видит «дефицит» на день
- Хранение как TEXT[] не позволяет учитывать в LP

**Решение:**
- Компенсация **недельная**: пятничный круассан перераспределяется по всей неделе
- Хранение: `user_indulgence_items` с product_id + serving_g + day_of_week + meal_context
- LP учитывает индульгенции как fixed cost на неделю, остаток распределяет оптимально

**Последствия:**
- Новая таблица user_indulgence_items (не TEXT[] в профиле)
- Indulgence Planner — отдельный модуль
- UI-тон: нейтральный. «Учтено в плане.» Никаких оценок.

## ADR-014: Математическая модель Value Score (LP + Composite Score)

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** исходная спецификация — произвольные веса (w_nutriscore × A + w_price × B + w_deficit × C).

**Проблема с произвольными весами:**
- Нет RCT под конкретную весовую схему
- Нельзя гарантировать что корзина оптимальна по белку/бюджету
- При изменении весов результат непредсказуем

**Решение: два уровня математики:**

**Уровень 1 — LP для корзины:**
```
max Σ(composite_score × x)
s.t. Σ(price × x) ≤ budget
     Σ(protein × x) ≥ target_protein
     Σ(calories × x) ∈ [cal_min, cal_max]
     sacred_items ⊆ x
```
LP гарантирует математическую оптимальность при данных ограничениях.

**Уровень 2 — Composite Score для слотов:**
```
score = w_ns × norm(NS) + w_price × norm(1/₽_protein) + w_deficit × norm(deficit)
```
Веса берутся из таблицы по бюджету пользователя (не произвольны — зафиксированы как бизнес-правило).

**Честная оговорка:** composite_score — эвристика с дисклеймером. LP — математически корректный, но результат зависит от качества данных.

**Последствия:**
- Scoring Pipeline: scipy.optimize.linprog / PuLP
- value_scores.composite_score (не просто value_score)
- Дисклеймер в UI на Value Score экране
