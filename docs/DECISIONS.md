## ADR-001: NutriScore-2023 как основа Quality Gate

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** нужна система оценки качества продуктов для автоматической фильтрации. Варианты: собственная формула, Nutri-Score, NuVal, NOVA отдельно.

**Решение:** используем алгоритм NutriScore-2023 (версия обновлённая в 2023 году).

**Обоснование:** единственная система маркировки с серьёзной RCT-валидацией (n=21 159, OR=29.0). Второй независимый RCT (n>5000) подтвердил эффект. Не изобретаем своё — используем то, что прошло независимую валидацию.

**Последствия:**
- Нужен точный расчёт NutriScore по алгоритму 2023 года (не 2017)
- NOVA-классификация используется как дополнительный критерий (Umbrella review, но не RCT на исходах)
- Собственные «штрафные формулы» за E-шники убраны — не имеют RCT-базы у взрослых

## ADR-002: Юзкейс-первичные свопы (convenience_tier + use_context)

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** система предлагала замены по нутриентам (колбаса → куриная грудка сырая). Это бессмысленно — пользователь покупает колбасу не ради белка, а чтобы сделать бутерброд за 0 минут.

**Решение:** каждый товар получает `convenience_tier` (1/2/3) и `use_context` (sandwich, fry_with_egg, etc.). Свопы — только внутри одного слота.

**Обоснование:** замена работает только если закрывает тот же функциональный слот. «Behaviour substitution» как BCT имеет доказанную эффективность (SR), но только при сохранении контекста использования.

**Последствия:**
- Каждый товар в базе нужно разметить по convenience_tier и use_context
- Первичная разметка — LLM-assisted, затем ручная корректировка
- SQL-запросы подбора замен включают обязательный фильтр по tier и context

## ADR-003: Прогрессивная модель (не ребилд корзины)

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** первая версия предполагала сразу генерировать полную «правильную» корзину. Это провальная стратегия — пользователь не готов менять всё сразу.

**Решение:** 4-этапная прогрессивная модель (1 своп → 3 → полкорзины → автопилот) с пользовательским контролем темпа.

**Обоснование:**
- SR/MA (PMC 2024, 27 исследований): медиана формирования привычки = 59 дней, только 23% достигают автоматизма
- RCT по постепенному снижению сахара: инкрементальные изменения по ~10% не снижают удовольствие
- Wood & Neal: формирование новой привычки эффективнее запрета старой
- Graded Task Assignment как BCT имеет доказанную эффективность в диетарных интервенциях

**Последствия:**
- user_nutrition_profiles.progression_stage (1-4)
- Переходы между этапами только по явным триггерам (купил 2+ раза, нажал «хочу больше»)
- Никакого принудительного расписания, нет уведомлений с давлением

## ADR-004: Честные уровни уверенности ([RCT] / [SR/MA] / [i])

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** большинство food-приложений на рынке делают уверенные утверждения о здоровье без указания качества доказательств. Это создаёт ложное чувство научности.

**Решение:** каждое нутритивное утверждение в интерфейсе сопровождается уровнем уверенности:
- ✅ `[RCT]` — рандомизированные контролируемые исследования
- ⚠ `[SR/MA]` — систематические обзоры / мета-анализы (могут быть когортные данные внутри)
- ℹ `[i]` — обсервационные данные, эвристики, арифметика

**Обоснование:** приложение касается здоровья. Если оно будет давать уверенные рекомендации на слабой доказательной базе — это хуже, чем вообще ничего не рекомендовать. Калькулятор ₽/г белка — полезный инструмент, но не «доказательный». NutriScore — валидирован RCT. Разница принципиальна.

**Последствия:**
- UI: плашки с уровнем уверенности на каждой карточке
- Дисклеймер «не является медицинской рекомендацией» на всех нутритивных экранах
- docs/EVIDENCE_BASE.md — отдельный документ с полной доказательной базой

## ADR-005: CF как retention-инструмент, не нутритивная персонализация

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** коллаборативная фильтрация (CF) в стиле Spotify — мощный инструмент для рекомендаций. Но в контексте питания нужно честно определить её роль.

**Решение:** Taste Engine (CF) позиционируется как инструмент retention и удобства, НЕ как медицинская персонализация.

**Обоснование:**
- Review of RCTs (Food & Function, 2026) разделяет «высокую» и «низкую» персонализацию. CF — это «низкая» (паттерны похожих пользователей, не биомаркеры)
- «Высокая персонализация» (реальные биомаркеры) показывает значимо лучшие результаты на качество диеты
- Называть CF «персонализацией» в медицинском смысле — нечестно

**Последствия:**
- CF не влияет на Quality Gate (товар прошёл/не прошёл — объективный критерий)
- CF не является компонентом «научного» скоринга — это отдельная система для retention
- В UI: «Похожие тренеры берут» — социальное доказательство, не медицинская рекомендация
- Taste Score НЕ смешивается с NutriScore в один «финальный score»

## ADR-006: PostgreSQL + pgvector вместо отдельного vector DB

**Статус:** Принято
**Дата:** Март 2026

**Контекст:** для Taste Engine нужны и реляционные данные (события, профили), и векторное хранилище (эмбеддинги для CF). Варианты: PostgreSQL + pgvector, PostgreSQL + Pinecone/Weaviate, MongoDB.

**Решение:** PostgreSQL + pgvector в одной базе.

**Обоснование:**
- Данные слишком структурированы и связаны между собой для MongoDB
- pgvector даёт ANN-поиск (приближённые ближайшие соседи) за миллисекунды даже на миллионах пользователей
- Не нужен отдельный vector DB сервис на этапе MVP — pgvector достаточно
- Все данные в одной базе — проще consistency и операции

**Последствия:**
- vector(512) для user_taste_vectors.embedding
- IVFFlat индекс для ANN-поиска
- При масштабировании (>1M пользователей) — пересмотреть в сторону HNSW индекса или отдельного vector DB
